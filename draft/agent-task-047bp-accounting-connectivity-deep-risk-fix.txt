P47bp - Accounting Connectivity Deep-Risk Fix (Purchases/Orders/Expenses/Staff/Analytics)

Objective
- Remove high-risk accounting drift paths where operational status changes are not reflected in ledger or reports.
- Keep behavior consistent across purchase/expense/staff check flows.
- Ensure analytics and staff due/payable numbers are based on cleared funds, not raw pending checks.

Scope (only)
- src/app/api/check-passing/route.ts
- src/server/modules/expenses.ts
- src/app/api/staff/[id]/payments/route.ts
- src/app/api/staff/route.ts
- src/app/api/staff/[id]/route.ts
- src/server/modules/analytics.ts
- (if needed) src/server/modules/staff.ts or a new shared helper module under src/server/modules/

Do not
- Do not change business rules unrelated to accounting/check status.
- Do not add unrelated refactors.
- Do not include commit/push instructions.

--------------------------------------------------------------------
A) CRITICAL: Check-passing status update must trigger domain ledger reconciliation
--------------------------------------------------------------------

Problem
- `src/app/api/check-passing/route.ts` currently updates `checkStatus` directly for Purchase/Expense/Staff.
- This bypasses domain ledger/status logic (especially Purchase and any future Staff/Expense reconciliation rules).
- Result risk: check status changes but ledger/payment status stays stale.

Required fix
1) In `PATCH` flow, replace direct `tx.purchasePayment.update / tx.expense.update / tx.staffPayment.update` paths with domain-safe handlers:
   - Purchase: call existing `updateCheckStatusCore({ paymentId, status, user })`.
   - Expense: add and call a dedicated module function that updates status and re-runs expense ledger posting.
   - Staff: add and call a dedicated module function that updates status and re-runs staff payment ledger posting.

2) Keep CheckPassingItem refresh and CheckPassingLog creation, but ensure they happen after successful domain reconciliation.

3) Preserve bulk update capability (`updates[]`), but correctness first:
   - sequential execution is acceptable.
   - if one update fails, return clear error mentioning failing source/id.

Acceptance
- From Check Passing UI, changing status must always update both source status and ledger state consistently.

--------------------------------------------------------------------
B) CRITICAL: Expense settlement ledger must ignore uncleared check amount
--------------------------------------------------------------------

Problem
- In `postExpenseLedger` (`src/server/modules/expenses.ts`), settlement currently posts full `amount` whenever `isPaid=true`.
- Pending check should not credit bank/cash yet.

Required fix in `postExpenseLedger`
1) Compute:
   - `amount = expense.amount`
   - `checkAmount = clamp(expense.check, 0, amount)`
   - `cashPortion = expense.isPaid ? max(0, amount - checkAmount) : 0`
   - `passedCheckPortion = expense.isPaid && checkAmount > 0 && expense.checkStatus === 'Passed' ? checkAmount : 0`
   - `settlementAmount = cashPortion + passedCheckPortion`

2) Keep primary accrual posting unchanged (Dr Expense / Cr AP) for approved expenses.

3) Settlement posting (`expenseSettlement:<id>`) behavior:
   - create entries only if `settlementAmount > 0`
   - entries:
     - Dr AP `settlementAmount`
     - Cr paidFromAccount `settlementAmount`
   - pending/bounced/cancelled check part must not be credited to cash/bank.

Acceptance
- Expense with full pending check: no settlement entries.
- Expense check switched to Passed: settlement entries appear correctly.
- Switching back to Pending/Bounced/Cancelled removes check-settlement effect.

--------------------------------------------------------------------
C) CRITICAL: Staff payment check handling must use cleared-funds ledger logic
--------------------------------------------------------------------

Problem
- `src/app/api/staff/[id]/payments/route.ts` currently posts full amount to ledger at create time, even when check is Pending.
- This overstates cleared cash/bank movement.

Required fix
1) Extract staff payment ledger posting into a reusable function (same file or module):
   - input: payment row + transaction client
   - postingGroup: `staffPayment:<paymentId>`
   - clear existing group before rebuild
   - use cleared-funds formula:
     - `checkAmount = clamp(check, 0, amount)`
     - `cashPortion = max(0, amount - checkAmount)` (immediate)
     - `passedCheckPortion = checkStatus === 'Passed' ? checkAmount : 0`
     - `ledgerAmount = cashPortion + passedCheckPortion`
   - create Dr Salaries Payable / Cr source account only for `ledgerAmount > 0`.

2) POST create flow:
   - Create staffPayment.
   - Create/update CheckPassingItem for check payments.
   - Call the reusable ledger rebuild function (so pending check creates zero settlement).

3) Check-passing Staff status update flow:
   - After status change, call the same reusable ledger rebuild function.

Acceptance
- Staff check Pending: no cash/bank credit for check portion.
- Staff check Passed: ledger becomes Dr Salaries Payable / Cr source account for check portion.
- Staff check Bounced/Cancelled/Pending rollback removes uncleared check settlement.

--------------------------------------------------------------------
D) HIGH: Staff due/paid totals must exclude pending checks
--------------------------------------------------------------------

Problem
- Staff totals in `src/app/api/staff/route.ts` and `src/app/api/staff/[id]/route.ts` currently sum raw `staffPayment.amount`.
- Pending checks are treated as paid, causing due under-report.

Required fix
1) Add shared helper (server-side) to compute effective paid amount:
   - `effectivePaid = cashPortion + passedCheckPortion`
   - same formula as section C.

2) Update staff listing and single-staff endpoints:
   - Replace aggregate raw sums with effective cleared sum logic.
   - Keep payment history raw values visible, but totals/due must use effective paid.

Acceptance
- Pending check payment must not reduce due.
- Passing the check must reduce due.

--------------------------------------------------------------------
E) HIGH: Analytics expenses must include staff payouts consistently
--------------------------------------------------------------------

Problem
- `src/server/modules/analytics.ts` assumes staff payments are already in `Expense`.
- Current implementation does not reliably enforce that linkage.

Required fix
1) Add staff payment effective-cleared aggregation and merge into analytics expense totals.
2) Avoid double counting:
   - include staff payments where no linked Expense exists (if relation is available).
3) Apply same logic to:
   - summary expenses/netProfit
   - monthly breakdown expenses/profit

Note
- Keep ad expense (`isAdExpense`) logic unchanged.
- Keep order snapshot revenue/cogs/courier logic unchanged.

Acceptance
- Staff payouts (cleared) affect analytics expenses even when no Expense row exists.
- Pending staff checks do not inflate expenses as paid cashflow.

--------------------------------------------------------------------
F) MEDIUM: Expenses page summary currently overstates paid when checks are pending
--------------------------------------------------------------------

Problem
- In `getExpenses(... includeTotal=true)` (`src/server/modules/expenses.ts`), `totalPaid` is derived from `isPaid=true`.
- For check payments, `isPaid=true` with `checkStatus='Pending'` still appears as fully paid in summary.

Required fix
1) Replace current `groupBy isPaid` paid/unpaid summary with cleared-funds summary logic:
   - `clearedPaid` = cash portion + passed check portion
   - `uncleared` = amount - clearedPaid
2) Keep backward-compatible response keys:
   - `totalPaid` should become cleared paid amount.
   - `totalUnpaid` should reflect uncleared amount.

Acceptance
- Pending check expenses increase `totalUnpaid` until passed.
- After pass, same amount moves into `totalPaid`.
