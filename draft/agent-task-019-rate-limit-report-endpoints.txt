UPDATE P19a - Add rate limiting to heavy report endpoints (safe)

Scope: ONLY apply rate limit checks. No logic refactors.

Goal:
- Protect heavy report endpoints from bursts (per staff ID/IP).
- Use existing checkRateLimit util (Redis-backed, memory fallback).
- Return 429 on limit exceeded.

Rate limits (recommended):
- analytics: 60 req / 60s
- marketing overview + campaigns list: 60 req / 60s
- accounting ledger: 30 req / 60s
- accounting balance sheet: 30 req / 60s

1) Analytics
File: src/app/api/analytics/route.ts
- Import: checkRateLimit from '@/server/utils/rate-limit'.
- After enforcePermission success:
  const staffId = (allowed staff)?.id; // from enforcePermission
  const key = `analytics:${staffId || req.headers.get('x-forwarded-for') || 'anon'}`;
  const ok = await checkRateLimit(key, 60, 60);
  if (!ok) return NextResponse.json({ error: 'Too many requests' }, { status: 429 });
