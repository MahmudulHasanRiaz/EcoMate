P47da - Woo Sync imports only 8 orders: fix worker cache crash + truthful sync metrics

Context (confirmed from production logs)
- DB check shows only 8 Woo orders in last 10 days:
  - source='woo' count = 8
- Worker logs show repeated hard failure on every batch item:
  - "Invariant: incrementalCache missing in unstable_cache ... appSetting.findUnique({ key: 'general' })"
- Root cause:
  - `src/server/utils/app-settings.ts` uses `unstable_cache`.
  - BullMQ worker runtime has no Next incremental cache.
  - So `getGeneralSettings()` crashes in worker when Woo batch sync runs.
- Side issue:
  - manual sync result shows inflated "updated" counts because fetched batch length is counted, not successful imports.

Goal
- Make Woo sync work in worker runtime (no incrementalCache crash).
- Make sync response truthful (fetched/imported/skipped/failed/queued).
- Keep plugin test connection behavior untouched (already OK).

Scope
- src/server/utils/app-settings.ts
- src/server/modules/woo-sync.ts
- src/app/api/orders/import/woo/route.ts
- src/app/dashboard/settings/integrations/page.tsx

No unrelated refactor.

--------------------------------------------------
1) Runtime-safe settings access (critical)
--------------------------------------------------
File: src/server/utils/app-settings.ts

Implement a safe fallback path for worker runtime.

Required shape:
- Extract pure DB readers:
  - `readBrandingSettingsFromDb()`
  - `readGeneralSettingsFromDb()`
- Keep existing `unstable_cache` wrappers for web runtime speed.
- Exported functions must NEVER throw incrementalCache error:
  - `getBrandingSettings()`
  - `getGeneralSettings()`

Behavior:
- Try cached function first.
- If error message includes "incrementalCache missing in unstable_cache", fallback to direct DB reader.
- Re-throw other errors.

Pseudo pattern:
- `try { return await cachedReader(); } catch (e) { if (isIncrementalCacheMissing(e)) return await readFromDb(); throw e; }`

This keeps Next caching for pages/layout, and keeps worker stable.

2) Woo sync metrics must be truthful (critical)
--------------------------------------------------
File: src/server/modules/woo-sync.ts

2.1 Change batch processor to return summary
- Update `syncWooOrdersBatch(batch, integration)` to return:
  - `{ fetched: number, imported: number, skipped: number, failed: number, failedIds: string[] }`
- For each order:
  - `syncOneWooOrder` returns action imported/skipped -> increment accordingly.
  - Catch -> increment failed + push id.

2.2 Fix importWooOrders counters
- In `importWooOrders` maintain:
  - `fetched`
  - `queued`
  - `imported`
  - `skipped`
  - `failed`
  - `failedIds`
- NEVER do `updated += batch.length` as success metric.
- If queued path is used:
  - increment `queued += batch.length`
  - do not increment imported/skipped/failed in that branch.
- If inline path used:
  - aggregate from `syncWooOrdersBatch` return.

2.3 Return payload
- Return:
  - `{ fetched, queued, imported, skipped, failed, failedIds, fetchedPages, totalPages }`
- Keep backward compatibility fields temporarily if needed:
  - `updated` can map to `imported` only (not fetched).
