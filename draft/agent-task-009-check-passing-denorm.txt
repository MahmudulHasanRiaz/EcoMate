UPDATE P04c - Check Passing denormalized table (replace UNION query)

Scope: Implement a persistent `CheckPassingItem` table and update flows to keep it in sync.
Goal: Remove heavy UNION query in `api/check-passing` and speed up search/filter.

Decisions:
- Keep “Pending” checks as Due (no separate post‑dated account).
- No delete of financial data; only maintain a derived table.
- Use real‑time write‑time sync + a rebuild script.

1) Prisma schema
File: prisma/schema.prisma
Add enum + model:
```
enum CheckPassingSource {
  Purchase
  Expense
  Staff
}

model CheckPassingItem {
  id             String   @id @default(cuid())
  source         CheckPassingSource
  sourceId       String
  passingDate    DateTime
  amount         Float
  status         CheckStatus
  checkNo        String?
  referenceId    String?
  referenceLabel String?
  referenceUrl   String?
  payee          String?
  type           String?
  createdAt      DateTime
  updatedAt      DateTime

  @@unique([source, sourceId])
  @@index([passingDate, status])
  @@index([source, sourceId])
  @@index([checkNo])
  @@index([referenceLabel])
  @@index([payee])
}
```
Run migration:
```
npx prisma migrate dev --name add_check_passing_items
npx prisma generate
```

2) Helper module
File: src/server/modules/check-passing-items.ts (new)
Export helpers:
- `upsertCheckPassingItem(tx, payload)` (tx = prisma or transaction)
- `deleteCheckPassingItem(tx, source, sourceId)`
- `buildCheckPassingItemFromPurchasePayment(...)`
- `buildCheckPassingItemFromExpense(...)`
- `buildCheckPassingItemFromStaffPayment(...)`

Each builder must:
✓ return null if check amount <= 0 or checkDate is null.
✓ map `status` to CheckStatus (default Pending).
✓ include referenceLabel/referenceUrl/payee/type same as current UNION query.

3) Wire write‑time sync (critical)
Purchase payments:
- File: src/server/modules/purchases.ts
- After create/update payment record in:
  - upsertPurchasePaymentCore
  - addPurchasePaymentCore
  - updateCheckStatusCore
  - deletePurchasePaymentCore (delete derived item)
Call helper:
```
const item = await buildCheckPassingItemFromPurchasePayment(tx, paymentRecord.id);
if (item) await upsertCheckPassingItem(tx, item);
else await deleteCheckPassingItem(tx, 'Purchase', paymentRecord.id);
```
Implementation detail: builder can fetch needed joins (po, supplier, vendor, productionStep).

Expenses:
- File: src/server/modules/expenses.ts
- After create/update expense, call builder.
- On delete, remove item.

Staff payments:
- File: src/server/modules/staff.ts (or wherever staff payments are created/updated)
- After create/update staff payment, call builder.
- On delete, remove item.

4) Update Check Passing API to read from CheckPassingItem
File: src/app/api/check-passing/route.ts
Replace UNION query with:
```
FROM "CheckPassingItem"
```
Keep existing cursor + filters (passingDate/status/source/search/checkNo).
Search should still check referenceLabel/referenceId/payee/type/checkNo (same as now).

5) Backfill script (one-time)
File: scripts/rebuild-check-passing.ts (new)
- Fetch all PurchasePayment/Expense/StaffPayment with check > 0 and checkDate != null
- Build items and upsert into CheckPassingItem
- Optional: clear table before rebuild (`deleteMany({})`)
Run once after migration:
```
npx tsx scripts/rebuild-check-passing.ts
```
