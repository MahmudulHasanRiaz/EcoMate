1) Order Details status-save flow hardening (no backend blocker currently observed)
File:
- `src/components/orders/order-details-view.tsx`

Observed:
- Backend status update works (PATCH 200 seen in logs).
- UI still has fragile form wiring that can silently no-op in edge cases.
- `statusForm` starts with `status: undefined`.
- After `orderData` loads, form is never reset with current status.
- Status `<Select>` is wired as uncontrolled (`defaultValue={field.value}`) instead of controlled (`value={field.value}`).
- Risk: payload may omit `status` if save is clicked without explicit selection.

- When order data is fetched, call:
  - `statusForm.reset({ status: orderData.status, officeNote: orderData.officeNote || '' })`
- Change status select binding to controlled:
  - `value={field.value ?? ''}`
  - `onValueChange={field.onChange}`
- Keep save button as explicit submit trigger, but prevent empty-status submission:
  - If status is empty, show destructive toast and return before API call.
- In success toast, use the server-returned status (`updated.status`) instead of raw form value.

Acceptance:
- Status dropdown preloads current order status.
- Changing status + Save updates immediately.
- Save cannot silently no-op because of empty/undefined status.
