P47p - Treat CANCELLED as terminal (TTL guard hardening)

Goal
- If a lead was manually cancelled, do not reopen it immediately from new incomplete pings.
- Use the same RECENT_COMPLETION_TTL_SEC window.

Scope
- ONLY: src/app/api/woo/incomplete-orders/route.ts

Change
1) Extend terminal statuses to include CANCELLED

File: src/app/api/woo/incomplete-orders/route.ts
Function: hasRecentCompletion(...)

Find this block:

```ts
// DB fallback: recent terminal lead (completed or converted)
const recentCompletedLead = await prisma.wooCheckoutLead.findFirst({
  where: {
    integrationId,
    phoneNormalized: phone,
    status: { in: ['COMPLETED', 'CONVERTED'] },
    OR: [
      { completedAt: { gt: cutoff } },
      { convertedAt: { gt: cutoff } },
    ],
  },
  select: { id: true },
});
```

Replace with:

```ts
// DB fallback: recent terminal lead (completed, converted, or cancelled)
const recentCompletedLead = await prisma.wooCheckoutLead.findFirst({
  where: {
    integrationId,
    phoneNormalized: phone,
    status: { in: ['COMPLETED', 'CONVERTED', 'CANCELLED'] },
    OR: [
      { completedAt: { gt: cutoff } },
      { convertedAt: { gt: cutoff } },
    ],
  },
  select: { id: true },
});
```

Notes
- CANCELLED already sets completedAt in the PATCH route, so this will work without schema changes.
- This is intentionally TTL-based, not permanent. After TTL, a new incomplete can open again.

Test checklist
- Cancel a lead from the UI.
- Immediately go back to checkout with the same phone and leave the page.
  - Expected: API returns status=skipped_recent_completion.
  - Expected: lead does not reopen.
- After ~30 minutes (TTL), new incomplete pings may open again (expected behavior).
