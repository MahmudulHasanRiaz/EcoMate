P47 Review – What is done vs. wrong (please fix before plugin)

Summary of actual changes:
- Added Woo API endpoints: /api/woo/plugin-config, /api/woo/restriction-check, /api/woo/incomplete-orders, /complete (with API key auth).
- Added /api/restrictions with GET/POST/DELETE (now permission‑guarded).
- Added IP/phone block/unblock buttons in Order + Customer pages.
- Added integration settings fields in settings page + business phone/address fields.
- Added phone-utils + Woo auth helper.

Critical issues / mismatches
1) Phone normalization is wrong vs requirement
   - File: src/lib/utils/phone-utils.ts
   - Currently only accepts 01 / 8801 / 10-digit starting with 1.
   - Must accept any 11‑digit starting with 0 (0XXXXXXXXXX), and 880/+880 + 10 digits (normalize to 0 + 10 digits). Do NOT require 01.
   - If invalid: return null → restriction-check should block + show message, incomplete capture should reject (no lead).

2) IP shown in UI (not allowed)
   - File: src/app/dashboard/orders/[id]/page.tsx
   - It prints raw IP and uses it in UI.
   - Requirement: IP must NOT be displayed. Only show “Blocked/Unblocked” status + action. Hash only in DB.

3) Manual block logic does not respect scope/integration
   - Order restriction model requires `scope` and (for site scope) `integrationId`.
   - Current UI blocks via /api/restrictions with no scope; API defaults SITE with integrationId null → phone blocks won’t match site checks.
   - Fix: when blocking phone from customer page, decide scope (SITE vs GLOBAL). If SITE, must provide integrationId (use most recent order’s integrationId or allow user select). If GLOBAL, set scope=GLOBAL.
   - For IP block: MUST be GLOBAL (per requirement). Set scope=GLOBAL in API or per request when targetType=IP.

4) /api/woo/restriction-check lacks general‑settings fallback support phone
   - File: src/app/api/woo/restriction-check/route.ts
   - It only falls back to integration.supportPhone and ignores business/general settings.
   - Required fallback: restrictionSupportPhone → integration.supportPhone → business.phone → app general settings phone/supportPhone.

5) /api/woo/plugin-config returns validation regex but server validation disagrees
   - regex allows 0XXXXXXXXXX; phone-utils currently rejects those.
   - Must align: either update regex or update normalizeBdPhone (preferred: update normalizeBdPhone).

6) Incomplete capture does not use dedupe settings
   - File: src/app/api/woo/incomplete-orders/route.ts
   - It upserts by fingerprint always; ignores integration.dedupeMinutes/debounceMs/retrySeconds.
   - Requirement: dedupeMinutes should prevent new records and only update if within window. Debounce/retry should be returned and/or enforced.

7) Restriction on successful order placement is missing
   - Must create a phone restriction after successful order (scope SITE) using duration type (Always/Hours/Days). Message + support phone from integration settings.
   - Place in webhook order handler or in /api/woo/incomplete-orders/complete when called from successful order event (needs payload to confirm success).

8) Dead code / duplication
   - src/server/modules/woo-public.ts is unused (no imports). Either delete or refactor API routes to use it. Don’t keep two implementations.

Other notes
- /api/restrictions now permission-guarded (good), but uses orders permission. Consider integrations:manage if that’s the intended admin scope.
- /api/restrictions POST ignores `message` passed by UI; if message is intended to be custom per block, accept it, else remove from UI payload.
