AUDIT 02 - Purchase module polish (backend + accounting + scalability)

Scope: Fix the issues listed below only. Keep changes minimal. No refactors unless specified.

---
Critical Accounting Consistency
1) Ledger postings for payments are inconsistent (Inventory vs AP)
File: src/server/modules/purchases.ts
Function: upsertPurchasePaymentCore
Issue: Ledger entries debit Inventory, credit Cash. Should debit AP for payment.
Fix:
- Use Accounts Payable account instead of Inventory for payment ledger.
- Align with addPurchasePaymentCore (Dr AP, Cr Cash/Bank).
- Keep postingGroup = `purchasePayment:${paymentRecord.id}` for idempotency.

2) Payment updates create unreconciled ledger entries
File: src/server/modules/purchases.ts
Functions: createPurchaseOrderCore, addPurchasePaymentCore, deletePurchasePaymentCore
Issue:
- createPurchaseOrderCore writes ledger entries for initial payments using sourceTransactionId = poId (no postingGroup).
- deletePurchasePaymentCore does not reverse ledger entries.
Fix:
- When creating PO with initial payments: create payment records first, then create ledger entries per payment with postingGroup = `purchasePayment:${paymentId}`.
- deletePurchasePaymentCore should reverse or delete ledger entries by postingGroup (or create reversing entries) when a payment is deleted.
- Ensure no duplicate ledger entries remain after payment edit/delete.

3) Check handling should not count as paid until Passed
File: src/server/modules/purchases.ts
Functions: syncStepPaidFromPaymentsTx, recomputePaymentStatusTx, updateCheckStatusCore
Issue: Pending checks count as paid; check status updates do not recompute payment status.
Fix:
- Only include check amounts when checkStatus === 'Passed' (Cash is always counted).
- After updateCheckStatusCore, call syncStepPaidFromPaymentsTx + recomputePaymentStatusTx.
- If checkStatus changes to Bounced/Cancelled, create reversing ledger entries (see item #2).

4) Clearing check should clear date/status
File: src/server/modules/purchases.ts
Function: upsertPurchasePaymentCore
Issue: If check is set to 0, checkDate/checkStatus may remain from old value.
Fix:
- When check is 0, set checkDate = null and checkStatus = null.

5) Internal fabric should not create AP liability
Files: src/server/modules/purchases.ts, src/server/modules/accounting.ts
Issue: Internal fabric is own inventory; AP should not be credited.
Best-practice fix (chosen): Use Work-In-Progress (WIP) asset account.
Fix:
- Add default account: "Work In Progress" (Asset) in DEFAULT_ACCOUNTS.
- Add ACCOUNT_LABELS.wip = 'Work In Progress'.
- Internal fabric consumption: Dr WIP, Cr Inventory (raw fabric). No AP.
- Printing/Cutting costs: Dr WIP, Cr AP (vendor payable).
- Final receiving: Dr Inventory (finished), Cr WIP for total production cost.
- Ensure WIP balance clears to 0 for a fully received PO.

6) Internal fabric POs show Unpaid at create
File: src/server/modules/purchases.ts
Function: createPurchaseOrderCore
Issue: paymentStatus computed from payments only; internal fabric should be treated as Paid (no due).
Fix:
- If usingInternal, set paymentStatus to Paid (or call recomputePaymentStatusTx after create).
- Avoid AP requirement check when usingInternal.

8) Internal fabric allocations round yards to integers (precision loss)
File: src/server/modules/purchases.ts
Functions: createPurchaseOrderCore, updateThreePieceFabricPlanningCore
Issue: Math.round on yards drops decimals.
Fix:
- Remove Math.round; keep decimal (Number(alloc.yards) || 0).
- Clamp to 2 decimals and use tolerance in allocation validation (abs(diff) <= 0.01).

9) External fabric revert uses non-atomic quantity update
File: src/server/modules/purchases.ts
Function: updateThreePieceFabricPlanningCore (external flow)
Issue: Uses data: { quantity: lot.quantity + delta } which is race-prone.
Fix:
- Use { quantity: { increment: delta } }.

10) Received quantity mismatch risk
File: src/server/modules/purchases.ts
Function: receivePurchaseOrderStockCore
Issue: finalReceivedQty uses payload "quantity" even when item-level quantities differ.
Fix:
- Compute totalReceived from item quantities (receivedItems or item.finalQty) and store that instead.
- Validate payload quantity matches sum, or ignore payload quantity.
