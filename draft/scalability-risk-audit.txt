৩) Purchases মডিউল (Deep Learn Notes)
- General vs 3‑piece split: Prisma PurchaseType (`general`, `three_piece`) এবং UI mapping আছে।
- 3‑piece flow: Production steps (FABRIC/PRINTING/CUTTING/FINISHING) + per‑step payments.
- Internal fabric usage: FabricLotUsage ভিত্তিক lot allocations, allocation ↔ unit cost sync, usage validation (yards must match) আছে।
- Payment status recompute: production payments + general payments + internal fabric adjustment যোগ করে status recalculation আছে।
- Offline invoice: PurchaseOrder.offlineInvoiceUrl এবং per‑step invoiceUrl update; UI তে View/Replace/Upload সেকশন আছে।

Risks/Edge‑cases in Purchases
- List endpoint (`getPurchases`) full include করে; list‑view heavy হতে পারে।
- Stats endpoint full scan করে (findMany + loop) — aggregate query ভালো হবে।
- addPurchasePaymentCore এ paidFromAccountId null হলে ledger credit account null হতে পারে (non‑null assertion)।
- Internal fabric: lot cost resolution relies on PO lotNumber mapping; missing cost হলে 0 carry করতে পারে।

৪) অন্যান্য স্কেলিবিলিটি/প্রডাকশন রিস্ক (Plan‑এর বাইরেও)
- Orders search ILIKE/contains without pg_trgm index → বড় ডেটায় slow.
- Orders list: every call `count()` চালায়; 5–6k/day scale এ costly.
- Partners overview: `getPartnerOverviewStats` সব PO + relations টেনে aggregate করছে → heavy memory/CPU.
- Analytics monthly breakdown: প্রতি মাসে aggregate query loop (12x2) → cache/rollup দরকার।
- Expenses list: API সব expenses ফেরত দিচ্ছে → UI in‑memory pagination; বড় ডেটায় crash risk.
- Issues list: offset pagination + total count heavy; cursor + minimal select better।
- Staff list: Clerk avatar API প্রতিবার hit → latency + rate limit ঝুঁকি।
- Inventory client page: lots/movements on‑demand fetch wiring অসম্পূর্ণ হলে UI data blank হতে পারে (functional + UX risk)।
- Check passing summary: union query every request; materialized/cached summary নেই।
- Job queue: worker না চললে SMS/notifications silently pending থাকবে — production SOP দরকার।

৫) প্ল্যান‑এর সম্ভাব্য গ্যাপ/ঝুঁকি
- সব heavy রিপোর্টকে cache করলে freshness vs accuracy trade‑off হবে—business-critical metrics নির্দিষ্ট করতে হবে।
- Cursor pagination চালু করলে UI‑তে total count না দেখালে business users confusion হতে পারে (UI পরিবর্তন প্রয়োজন)।
- Partitioning/archival strategy পরিকল্পনায় আছে কিন্তু বাস্তবায়নের ডাটা মডেল, constraint, retention rule স্পষ্ট নয়।

৬) আপনার কাছ থেকে কনফার্মেশন দরকার
- আপনি কোন কোন মডিউল/ফাইলে ম্যানুয়াল পরিবর্তন করেছেন? (file list দিলে diff যাচাই সহজ হবে)
