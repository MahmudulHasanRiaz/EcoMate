Context
- UI console shows: `Failed to fetch delivery report from internal API` from `src/services/delivery-score.ts`.
- Ops requirement: Woo import must be strictly `processing` only. `on-hold`, `completed`, `cancelled`, `pending` must not be imported even if requested manually.

Scope (strict)
- src/services/delivery-score.ts
- src/app/api/orders/import/woo/route.ts
- src/server/modules/woo-sync.ts

Do not touch unrelated files.

--------------------------------------------------
1) Fix noisy delivery-report console error
--------------------------------------------------
File: src/services/delivery-score.ts

Required changes:
1. Normalize input phone before request using `normalizeBdPhoneForStorage`.
2. If normalized phone is empty or length < 11, return `null` without API call.
3. Use URLSearchParams for query string encoding.

   - log only `console.warn` for unexpected statuses (5xx, 429, etc.).

Acceptance:
- No `console.error('Failed to fetch delivery report from internal API')` remains.
- Expected non-OK responses do not flood browser console as errors.

2) Enforce Woo import to processing only (server-side hard guard)
--------------------------------------------------
Files:
- src/app/api/orders/import/woo/route.ts
- src/server/modules/woo-sync.ts

Required changes:
1. Route-level guard:
   - Always pass `status='processing'` to `importWooOrders`.
   - If request body provides different status, ignore it and log one warning.

2. Module-level guard:
   - In `importWooOrders`, normalize/force status filter to `processing` always.
   - If caller passes any non-processing status, ignore it and log warning.
   - Keep existing logic unchanged otherwise.

Acceptance:
- Any caller (UI/API/script) trying `on-hold` still fetches Woo API with `status=processing`.
- Existing push-to-hold behavior after import remains unchanged.
