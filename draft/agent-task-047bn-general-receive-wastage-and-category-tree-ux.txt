1) General receive: add per-item wastage support
--------------------------------------------------------------------

Business rule:
- For general PO receive, user enters:
  - receivedNow = good stock entering inventory
  - wastageNow = damaged/lost units (never enters inventory)
- Completion uses coverage:
  - receivedQty + generalWastageQty >= orderedQty

Data model:
A) Add field on PurchaseOrderItem:
- generalWastageQty Int @default(0)

B) Migration:
- Add non-null column with default 0.

Server changes (receivePurchaseOrderStockCore):
A) Extend payload item shape:
- from: { itemId, quantity, pindaBreakdown? }
- to: { itemId, quantity, wastageQty?, pindaBreakdown? }

B) Per-item validation:
- quantity >= 0
- wastageQty >= 0
- quantity + wastageQty <= remainingForItem
- remainingForItem = orderedQty - (receivedQty + generalWastageQty)

C) Update counters:
- receivedQty += quantity
- generalWastageQty += wastageQty
- Inventory updates and movements remain based on quantity only.
- Wastage never creates InventoryItem quantity change.

D) PO status logic:
- Replace old completion check (receivedQty < orderedQty) with coverage check.
- All complete => Received, else PartialReceived.

E) Logging:
- Include in PurchaseOrderLog:
  - current transaction receive total
  - current transaction wastage total
  - cumulative coverage total

Mapping/types:
- Add generalWastageQty in mapDbPoToAppPo purchase item mapping.
- Add field in src/types/index.ts purchase item type.

2) Cost model (MUST): PO created bill stays fixed, damage cost is redistributed
--------------------------------------------------------------------

Required behavior:
- po.total from creation is authoritative and must not change.
- Final total value of all good received units must equal po.total.
- Damaged units increase effective unit cost of remaining good units.

Implementation strategy (safe + deterministic):

A) During partial receive:
- Keep current behavior (use base unitCost for provisional inventory entries).

B) On transition to Received (all lines fully covered), run final reallocation in same transaction:
1. For each item i:
   - goodQty_i = receivedQty_i
   - baseUnit_i = item.unitCost
2. Compute weighted base value of good units:
   - baseGoodValue = SUM(goodQty_i * baseUnit_i)
3. Guard:
   - If baseGoodValue <= 0, throw error (cannot finalize with zero good units).
4. Allocate final bill by weighted share:
   - allocated_i = po.total * (goodQty_i * baseUnit_i / baseGoodValue)
5. Final unit cost per item:
   - finalUnit_i = allocated_i / goodQty_i (for goodQty_i > 0)

C) Rounding rule (important):
- Round finalUnit_i to project money precision.
- Ensure exact reconciliation to po.total:
  - Recompute final allocated sum.
  - Apply residual difference to the largest good-value line (or last eligible line).

D) Apply revaluation:
- Update PO-related inventory lot costs to finalUnit_i.
- Match lots by productId + variantId + PO lot signature (existing lot rule).
- Write InventoryMovement entries with type Adjusted and quantityChange = 0.
- Note should include: old unit cost, new unit cost, PO id, reason = general receive wastage redistribution.

E) Accounting safety:
- Do not mutate po.total.
- Do not add extra payment ledger postings in this cost normalization step unless existing valuation flow explicitly requires it.

Expected result:
- Final value of received good units equals original PO bill.
- If wastage rises, unit cost rises proportionally across non-damaged received goods.
