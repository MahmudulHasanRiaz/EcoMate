P47bz - Customer phone block: explicit site scope selection (clear UX, no ambiguity)

Goal
- Remove ambiguity of “This Site” on customer profile phone block.
- Ensure SITE scope always maps to a specific Woo integration/site (explicitly chosen).
- Keep UI simple, lightweight, and responsive.

Scope
- `src/app/dashboard/customers/[id]/page.tsx`
- `src/app/api/restrictions/route.ts`
- (optional helper use) `src/services/integrations.ts`

Out of scope
- No DB schema/migration
- No plugin code changes
- No unrelated refactor

--------------------------------------------------------------------
Problem
--------------------------------------------------------------------

Current behavior:
- If user chooses SITE, code silently uses `customerOrders[0]?.integrationId` (latest order).
- This is ambiguous and can target wrong site for multi-site customers.

Expected behavior:
- SITE must mean: a specific integration/site selected by user in the dialog.

--------------------------------------------------------------------
1) Backend guard (must)
--------------------------------------------------------------------

File: `src/app/api/restrictions/route.ts`

In POST handler, add strict validation:
- If `targetType === 'PHONE'` and `scope === 'SITE'`, `integrationId` is required.
- If missing -> return 400 with clear message:
  - `"integrationId is required for SITE phone restriction"`

Also validate integration ownership:
- If `integrationId` provided, ensure integration exists.
- If not found -> return 404.

This prevents accidental SITE blocks without a concrete site.

--------------------------------------------------------------------
2) Customer page UI: make site explicit (must)
--------------------------------------------------------------------

File: `src/app/dashboard/customers/[id]/page.tsx`

A) Add site options state derived from customer orders
- Build unique list of integrations from `customerOrders` where `integrationId` exists.
- Shape:
  - `{ integrationId, label }`
- Label priority:
  1. matched Woo integration `storeName` (if fetched)
  2. fallback: `${order.businessName || 'Business'} • ${order.platform || 'Website'}`
  3. final fallback: `Integration <short-id>`

B) Optional enrichment (recommended)
- Load Woo integrations via `getWooCommerceIntegrations()`.
- Map integrationId -> storeName for better labels.
- Keep this fetch lightweight and cached (single load on page).

C) Replace wording
- Change “This Site Only” label to “Specific Site”.
- Keep “Global” option.
- Add helper text under selector:
  - Global: block across all connected sites
  - Specific Site: block only for selected Woo site

D) Add explicit site picker (only when SITE selected)
- Show a compact Select/Combobox under scope buttons.
- Required only when `blockScope === 'SITE'`.
- Confirm button disabled until site selected.
- If no site options available:
  - show inline warning “No Woo site found from this customer’s orders”
  - disable SITE option and confirm fallback to GLOBAL only.

E) Submit payload
- On confirm:
  - `scope: 'SITE'` => send selected `integrationId`
  - `scope: 'GLOBAL'` => send `integrationId: null`
- Do not use `customerOrders[0]` as implicit site source anymore.

F) Success toast clarity
- For SITE:
  - `Phone blocked for <selectedSiteLabel>`
- For GLOBAL:
  - `Phone blocked globally`

--------------------------------------------------------------------
3) Restriction status check on customer page (recommended hardening)
--------------------------------------------------------------------

File: `src/app/dashboard/customers/[id]/page.tsx`

Current check uses optional integrationId from latest order.
For customer-level badge, use unscoped check:
- `checkRestriction(customer.phone, 'PHONE')` (no integrationId)
This reflects “blocked anywhere” status correctly.

Optional:
- keep small subtext showing active scopes count if you already fetch list.

--------------------------------------------------------------------
4) UX constraints (must keep)
--------------------------------------------------------------------

- No heavy modal redesign.
- Keep existing dialog layout.
- Mobile friendly:
  - scope buttons stacked cleanly
  - site select full width
  - clear helper copy

--------------------------------------------------------------------
5) Verification checklist
--------------------------------------------------------------------

A) Single-site customer
1. Open customer profile
2. Select Specific Site
3. Site auto-available and selectable
4. Block succeeds with integrationId present

B) Multi-site customer
1. Select Specific Site
2. Choose site A -> block
3. Confirm DB row uses integrationId of site A
4. Choose site B -> block independently (if needed)

C) No-site customer
1. SITE option should be disabled or show explicit warning
2. Cannot submit SITE block without selected site

D) API guard
1. Call POST `/api/restrictions` with `scope=SITE`, no integrationId
2. Expect 400

E) Regression
1. Global block still works
2. Unblock flow (`allScopes=true`) still works as before
