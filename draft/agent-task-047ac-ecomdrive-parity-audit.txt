P47ac - Ecomdrive parity audit (summary + coder actions)

Scope
- Reviewed draft/ecomdrive PHP + settings + assets list.
- Compared with Fashionary plugin (draft/fashionary-woo-plugin).

What Ecomdrive does (observed)
1) Validation architecture
- Dedicated validation core script (checkout-validation-core.min.js) loaded only on checkout.
- Centralized config localized via ecomdriveCheckoutConfig (includes mode flags, otp settings, enforcement mode).
- Server-side validation hooks on woocommerce_after_checkout_validation.

2) Enforcement modes
- Validation enforcement mode: normal/balanced/strict.
  - Balanced/Strict enforce server-side even if JS bypassed.
  - Strict is fail-closed when API/OTP missing.

3) Phone normalization
- Normalizes to 01XXXXXXXXX format; if 10 digits starting with 1 -> prefixes 0.
- Client + server consistency.

4) Bot / OTP
- Built-in OTP workflows, modal UI, and risk-based enforcement.
- Bot detection with thresholds (block/OTP) and honeypot fields.

5) Incomplete tracking resilience
- Settings mention fallback to worker/endpoint when server offline.
- JS is layered (core validation + tracking + optional modules).

6) UX
- Inline modals (error/otp/captcha), not just alert(); UI consistent on checkout.

What Fashionary plugin currently does
- Incomplete capture with phone validation + debounced beacons.
- Server-side restriction on checkout_process + after_checkout_validation.
- No client-side restriction-check on phone input (yet).
- Simple alert/inline error for invalid phone.

Primary gaps vs Ecomdrive
1) No client-side restriction-check to block immediately on phone input.
2) No enforcement mode (balanced/strict) if API unavailable.
3) No offline/local queue for incomplete leads when API is down.
4) No OTP/bot/anti-fraud stack (intentionally out of scope unless required).
5) UX: uses alert; no modal/overlay styling like ecomdrive.
