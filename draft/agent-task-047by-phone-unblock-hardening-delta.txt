--------------------------------------------------------------------
1) Make phone unblock idempotent in API
--------------------------------------------------------------------

File: `src/app/api/restrictions/route.ts`

Current behavior (allScopes phone delete):
- Returns 404 when `deleteMany.count === 0`

Required behavior:
- Always return 200 success for valid request.
- Response shape:
  - `{ success: true, deletedCount: number }`
- Keep 400 for invalid phone format.

Why:
- If restriction is already removed (another tab/user/action), unblock should still be treated as success.
- Prevents unnecessary “failed unblock” UX.

--------------------------------------------------------------------
2) Frontend: handle idempotent unblock cleanly
--------------------------------------------------------------------

File: `src/app/dashboard/customers/[id]/page.tsx`

Required changes:
1. Keep current `allScopes=true` DELETE call.
2. After DELETE response (even when deletedCount=0):
   - run `checkRestriction(customer.phone, 'PHONE', customerOrders[0]?.integrationId || undefined)`
3. Toast rules:
   - if deletedCount > 0: “Phone Unblocked”
   - if deletedCount === 0: “Already unblocked”
4. Only show destructive toast for true invalid request/server error (e.g., 400/500), not for zero-delete case.
