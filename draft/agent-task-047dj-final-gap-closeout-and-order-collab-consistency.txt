B) Multi-User Order Consistency Hardening (architectural, must implement)
==================================================

Goal:
- If one user updates an order, all other active users should see it quickly.
- Avoid write conflicts and silent overwrites during concurrent edits.

Implement in 4 focused layers:

1) Optimistic concurrency guard on order writes
Files:
- `src/lib/validations/orders.ts`
- `src/app/api/orders/[id]/route.ts`
- `src/server/modules/orders.ts`
- `src/services/orders.ts`

Required:
- Add optional `expectedUpdatedAt` (ISO string) in update payload schema.
- API PATCH passes it through to update module.
- In update module, before write:
  - compare current `updatedAt` with `expectedUpdatedAt`.
  - if mismatch => throw conflict error.
- API returns `409` with structured error:
  - `{ code: 'ORDER_CONFLICT', message, latest }`

Acceptance:
- Two simultaneous editors cannot silently overwrite each other.
- Second stale write gets 409 + latest snapshot.

--------------------------------------------------

2) Lock-aware mutation enforcement (server-side guard)
Files:
- `src/server/modules/order-open-lock.ts`
- `src/lib/validations/orders.ts`
- `src/app/api/orders/[id]/route.ts`
- `src/server/modules/orders.ts`
- `src/services/orders.ts`
- `src/components/orders/order-details-view.tsx`

Required:
- Add optional `lockToken` in update payload.
- Add helper in lock module to read current lock for an order.
- On PATCH write:
  - if a lock exists for another user and provided token does not match => return 409 `LOCKED`.
- When editing from order details:
  - send active lock token with update payload.

Acceptance:
- Locked order cannot be mutated by another user without valid lock token.
- Conflict is explicit instead of hidden overwrite.

--------------------------------------------------

3) Near-real-time cross-user refresh for order list/details
Files:
- `src/app/api/orders/changes/route.ts` (new)
- `src/server/modules/orders.ts` (helper query if needed)
- `src/app/dashboard/orders/client-page.tsx`
- `src/components/orders/order-details-view.tsx`

Required:
- New lightweight changes endpoint:
  - input: `since`, optional `ids`
  - output: `{ changedIds, serverTime }`
  - detect by `Order.updatedAt > since` (and optional id filter)
- Orders list page:
  - poll this endpoint every 5-8s (only when tab visible)
  - if changed ids intersect visible rows => `mutateOrders()`
- Order details page:
  - while open, poll specific order every 8-10s (pause while save in progress)
  - if changed externally, refresh view and show subtle toast/info.

Acceptance:
- User B sees User A's updates quickly without manual reload.
- No aggressive full-list polling every few seconds.
