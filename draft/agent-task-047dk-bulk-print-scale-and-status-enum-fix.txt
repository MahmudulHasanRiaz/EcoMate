1) Fix A - Status Enum Mismatch (RTS / internal values)
==================================================

Root cause:
- `ORDER_STATUSES` exposes internal statuses (`RTS__Ready_to_Ship_`, `In_Courier`, etc.).
- `orderStatusSchema` only accepts display labels (`RTS (Ready to Ship)`, `In-Courier`, etc.).
- Bulk/inline updates can send internal enum => `Validation failed`.

Files:
- `src/lib/validations/orders.ts`
- (if needed) `src/lib/order-statuses.ts`
- (optional normalization helper) `src/lib/order-status-normalize.ts` (new)

Required implementation:
- Make status validation accept BOTH forms:
  - internal: `Packing_Hold`, `In_Courier`, `RTS__Ready_to_Ship_`, `Return_Pending`, `Incomplete_Cancelled`
  - display: `Packing Hold`, `In-Courier`, `RTS (Ready to Ship)`, `Return Pending`, `Incomplete-Cancelled`
- Keep strict whitelist behavior (no arbitrary strings).
- Ensure both create and update schemas use this unified status validator.

Acceptance:
- `updateOrderSchema.safeParse({ status: 'RTS__Ready_to_Ship_' })` => success.
- `updateOrderSchema.safeParse({ status: 'RTS (Ready to Ship)' })` => success.
- Bulk `Confirmed -> RTS` works without validation error.

2) Fix B - Bulk Sticker/Invoice Print URL Limit
==================================================

Root cause:
- `selectedOrders.join(',')` passed in query string to `/print/bulk`.
- Query becomes too long (70-100+ often breaks, depending on infra limits).

Files:
- `src/app/dashboard/orders/client-page.tsx`
- `src/app/print/bulk/client-page.tsx`
- `src/app/api/print/bulk/route.ts` (new)
- `src/app/api/print/bulk/[token]/route.ts` (new) OR equivalent single route pattern
- `src/server/modules/print-batch.ts` (new, or use Redis utility directly)

Required implementation:
- Replace querystring `ids=` strategy with tokenized batch strategy:
  1. Client POSTs `{ type, ids }` to `/api/print/bulk`.
  2. Server stores ids by short token (Redis preferred) with TTL (e.g., 30 min).
  3. Server returns `{ token }`.
  4. Client opens `/print/bulk?type=sticker&token=<token>`.
  5. Print page loads ids via API by token, not from URL.
- Keep permission checks (`orders:read`) on retrieval.
- Add safe limit and UX message:
  - If ids > configurable threshold (e.g. 2000), reject with clear message.

Acceptance:
- 100+ selected orders no longer fails with Bad Request due URL size.
- 500+ opens print page reliably (network/DOM limits aside).
- No massive `ids=` query in opened URL.
