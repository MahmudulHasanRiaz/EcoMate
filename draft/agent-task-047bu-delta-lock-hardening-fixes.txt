3) CRITICAL: make lock acquire atomic (race-free)
--------------------------------------------------------------------

Problem
- Current `get` then `set` is race-prone; two users can acquire at once.

Required
1. In `src/server/modules/order-open-lock.ts`:
   - replace non-atomic read-then-write flow with atomic strategy.
   - use Redis Lua script (preferred) OR `WATCH/MULTI` safely.

Required behavior
- If key absent: acquire.
- If key exists with same `staffId`: refresh TTL and keep same token/openedAt.
- If key exists with different staff:
  - `force=false` => return `LOCKED` with current lock.
  - `force=true` => replace lock, return `overridden=true` and `previousLock`.

Implementation notes
- Do not use `uuid` package here. Use `crypto.randomUUID()` to avoid typing issues.
- Keep fail-open behavior when Redis unavailable.

Acceptance
- Concurrent acquire test: only one non-forced user acquires lock.

4) HIGH: permission hardening (prevent read-only lock hijack)
--------------------------------------------------------------------

Problem
- Any `orders.read` user can currently acquire/override locks and block editors.

Required
1. Route permissions:
   - `GET /api/orders/locks` => `orders.read` (keep)
   - `POST /api/orders/locks` => `orders.update`
   - `POST /api/orders/locks/heartbeat` => `orders.update`
   - `POST /api/orders/locks/release` => `orders.update`

2. Client handling:
   - if acquire returns 403, open details in read-only mode without lock attempts.
   - no override button for users without update permission.

Acceptance
- Read-only user cannot lock/override orders.

6) HIGH: lifecycle fix in `OrderDetailsView` (avoid re-acquire loop)
--------------------------------------------------------------------

Problem
- effect depends on mutable `lockStatus`, causing re-init/release noise.
- heartbeat interval too aggressive (10s).

Required
1. Split lock logic into separate effects:
   - effect A (mount/orderId change): acquire (if no external token).
   - effect B (token active): heartbeat every 20s.
   - cleanup: release only if this component owns lock.

2. Add explicit ownership flags:
   - `ownsLock` boolean.
   - if token passed from parent, set `ownsLock=false`.
   - parent-managed token should not be re-acquired.

3. Set heartbeat interval to 20 seconds.

Acceptance
- No repeated acquire calls from state feedback loop.
- Lock remains active while details open.
